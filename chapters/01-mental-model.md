# บท 1 — pwn คืออะไรกันแน่

## แก่นของ pwn ในหนึ่งประโยค

> **pwn คือการทำให้โปรแกรมทำในสิ่งที่ผู้เขียนไม่ได้ตั้งใจ โดยอาศัยการควบคุมหน่วยความจำ**

โปรแกรมทั่วไปทำงานตามที่ programmer สั่ง แต่ถ้ามี bug ที่ทำให้เรา
"เขียนทับ" ส่วนของหน่วยความจำที่ควบคุมการทำงานได้ เราก็บังคับให้มัน
ไปทำอย่างอื่นแทน เช่น เปิด shell ให้เรา

## ทำไมถึงเป็นไปได้

ภาษา C/C++ ให้ programmer จัดการหน่วยความจำเอง และ **ไม่ตรวจขอบเขต**
ให้ ถ้า buffer มีขนาด 64 ไบต์ แต่โปรแกรมรับ input 200 ไบต์ มันก็เขียน
เกินออกไปทับข้อมูลข้างๆ — นี่คือรากของช่องโหว่ส่วนใหญ่

```c
char name[64];
gets(name);   // รับ input ไม่จำกัดขนาด → เขียนเกิน 64 ไบต์ได้
```

## ภาพใหญ่: อะไรอยู่ตรงไหนในหน่วยความจำ

เมื่อโปรแกรมทำงาน หน่วยความจำถูกแบ่งเป็นโซนหลักๆ:

```
ที่อยู่สูง (high address)
┌─────────────────────┐
│       Stack         │  ← ตัวแปร local, return address (โตลงล่าง ↓)
│         ↓           │
│                     │
│         ↑           │
│        Heap         │  ← malloc() (โตขึ้นบน ↑)
├─────────────────────┤
│    BSS / Data       │  ← global / static variables
├─────────────────────┤
│    Text (Code)      │  ← โค้ดของโปรแกรม
└─────────────────────┘
ที่อยู่ต่ำ (low address)
```

สำหรับ pwn ระดับเริ่ม–กลาง เราสนใจ **Stack** เป็นหลัก เพราะ return
address (ตัวบอกว่าฟังก์ชันจบแล้วจะกระโดดกลับไปไหน) อยู่บน stack และ
ถ้าเราเขียนทับมันได้ = เราควบคุมได้ว่าโปรแกรมจะไปทำอะไรต่อ

## เป้าหมายสุดท้ายมักคือ RIP

**RIP** (Instruction Pointer) คือ register ที่ชี้ว่า CPU กำลังจะรัน
คำสั่งถัดไปที่ address ไหน ถ้าเราควบคุม RIP ได้ = เราสั่ง CPU ให้ไป
รันโค้ดที่เราต้องการได้

> เกือบทุกเทคนิคในเล่มนี้ (ret2win, ret2libc, ROP) คือวิธีต่างๆ ในการ
> **ควบคุม RIP แล้วชี้ไปที่โค้ดที่เป็นประโยชน์กับเรา**

## mitigation: ด่านที่ต้องผ่าน

ระบบสมัยใหม่มีการป้องกัน ทำให้ pwn ยากขึ้น ต้องรู้จักไว้:

| ชื่อ | ทำอะไร | ผลต่อเรา |
|------|--------|----------|
| **NX** | ห้ามรันโค้ดบน stack | ใช้ shellcode บน stack ไม่ได้ → ต้องใช้ ROP |
| **Canary** | วางค่าเฝ้าไว้ก่อน return address | ต้อง leak canary ก่อนถึงจะ overflow ผ่าน |
| **PIE** | สุ่มตำแหน่งโค้ดทุกครั้งที่รัน | ต้อง leak address ก่อนถึงจะรู้ว่าโค้ดอยู่ไหน |
| **ASLR** | สุ่มตำแหน่ง libc/stack/heap | ต้อง leak address ของ libc ก่อน |
| **RELRO** | ป้องกันเขียนทับ GOT | Full RELRO = เขียน GOT ไม่ได้ |

เราเช็ค mitigation ด้วยคำสั่ง [`checksec`](05-tools.md#1-checksec--ด่านแรกเสมอ) (บทที่ 5) แล้วเลือกเทคนิคตามนั้น

- โซนที่เราสนใจหลักคือ **Stack**
- เป้าหมายคือ **ควบคุม RIP**
- **mitigation** เป็นตัวกำหนดว่าจะใช้เทคนิคไหน
